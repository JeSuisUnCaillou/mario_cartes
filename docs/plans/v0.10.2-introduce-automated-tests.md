# Plan: Introduce E2E Tests for Game Lobby

## Context

The project has zero automated tests. We're introducing e2e tests starting with lobby behavior. Tests will boot a real Colyseus+Express server, connect real WebSocket clients, and assert on the full request/response flow.

## Stack

- **Vitest** as test runner
- **colyseus.js** client SDK (already in client workspace, add as server devDep)
- Native `fetch` (Node 24) for HTTP endpoint tests
- No `@colyseus/testing` — we boot the real server for true e2e coverage

## File structure

```
server/
  vitest.config.js
  tests/
    helpers.js          # startServer, createRoom, connectPlayer, waitForMessage...
    lobby.test.js       # all lobby e2e tests
```

## Test helper utilities (`server/tests/helpers.js`)

- `startServer()` — creates Express + Colyseus server on port 0, returns `{ baseUrl, cleanup }`
- `createRoom(baseUrl)` — calls `GET /create`, returns roomId
- `connectPlayer(baseUrl, roomId, opts)` — connects colyseus.js client, waits for `playerId` message, returns `{ room, playerId }`
- `connectBoard(baseUrl, roomId)` — connects with `{ type: "board" }`, returns `{ room }`
- `waitForMessage(room, type, timeout)` — wraps `room.onMessage` in a Promise
- `waitForPlayers(room, predicate, timeout)` — waits for a `players` broadcast matching a condition

Duplicates the ~10-line server setup from `index.js` (acceptable for test isolation — avoids modifying production code).

## Slices

### Slice 1: Test infrastructure + game creation

**Install:** `vitest` and `colyseus.js` as server devDependencies. Add `"test": "vitest run"` script.

**Tests:**
1. `GET /create` returns `{ id }` with a string id
2. `GET /find-or-create/:gameId` with new gameId creates room and returns `{ id: gameId }`
3. `GET /find-or-create/:gameId` with existing room returns `{ id: gameId }` (no duplicate)

### Slice 2: Joining a room + board client

**Tests:**
1. Player joins lobby — receives `playerId` message (string UUID)
2. Player appears in `players` broadcast with defaults: `cellId: 1, name: "???", ready: false, connected: true, handCount: 0`
3. Board client joins with `{ type: "board" }` — does NOT appear in players list

### Slice 3: Reconnection + page reload

**Tests:**
1. Player joins, disconnects (`room.leave()`), reconnects with same `playerId` — `connected` toggles false then true in players broadcast
2. Reconnected player receives `cardsDrawn` with their current card state
3. Board client disconnects and reconnects — receives `players`, `cellOccupants`, and `gameState` broadcasts (simulates page reload)
4. Player reconnects during playing phase — state fully restored (cards, position)

### Slice 4: Rejection after game started

**Tests:**
1. Two players join, both `setReady(true)`, game starts (phase -> "playing")
2. New player joins same room — receives `gameAlreadyStarted` message, does NOT appear in players list

### Slice 5: Player name

**Tests:**
1. `changeName("abc")` -> name becomes `"ABC"` in players broadcast
2. `changeName("a1b")` -> name becomes `"AB"` (digits stripped — regex is `/[^A-Z]/g`)
3. `changeName("ABCDEF")` -> name becomes `"ABC"` (truncated to 3)
4. `changeName("")` -> name stays `"???"`
5. Default name on join is `"???"`

### Slice 6: Ready state + game start

**Tests:**
1. `setReady(true)` -> player's `ready` becomes `true` in players broadcast
2. `setReady(false)` -> toggles back to `false`
3. Single player sets ready -> game starts (phase -> "playing" in `gameState` broadcast)
4. Two players, only one ready -> game does NOT start
5. Two players both ready -> game starts, both receive `cardsDrawn` with 5 cards
6. After game start: `currentRound` is 1, `activePlayerId` is set
