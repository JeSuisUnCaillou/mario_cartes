# Plan: Sequential card item processing (top to bottom)

## Context
Cards have ordered items (e.g. `["mushroom", "banana"]`). Currently the server batches all items ‚Äî accumulating moveCount, coins, and banana positions ‚Äî then resolves movement. Item order has no effect on gameplay. The user wants items to resolve **sequentially, top to bottom**, so `[mushroom, banana]` moves first then drops banana at the new cell, while `[banana, mushroom]` drops first then moves.

## Files to modify
- `server/rooms/GameRoom.js` ‚Äî main game logic
- `server/tests/lobby.test.js` ‚Äî update tests for new message format + add order tests
- `client/src/player.js` ‚Äî simplify `cardPlayed` handler, sequential throw animations

No changes needed to `client/src/board.js` ‚Äî its queue system already handles sequential broadcasts naturally.

## Slice 1: Server ‚Äî sequential item processing

### Add `pendingItems` to player state
In `_initialPlayerState()`, add `pendingItems: []`.

### Refactor `playCard` handler
Replace the batch accumulation loop with:
```js
player.discardPile.push(card);
player.pendingItems = [...card.items];
client.send("cardPlayed", { cardId: card.id, ...this._cardState(player) });
this._processNextItem(player);
```

### New `_processNextItem(player)` method
Shifts one item from `player.pendingItems`, resolves it, then recurses. Stops when:
- `pendingItems` is empty ‚Üí broadcast players (done)
- `pendingDiscard > 0` ‚Üí wait for discards
- `pendingShellChoice` ‚Üí wait for shell choice
- Player finished race ‚Üí clear pendingItems, end turn

### Extract resolution methods

**`_resolveMushroomStep(player)`** ‚Äî move 1 cell, broadcastCellOccupants, check lap, check banana/shell collision. On collision: `pendingDiscard = 1`, send `discardHit`.

**`_resolveBananaStep(player)`** ‚Äî drop banana at `player.cellId` (current position), broadcastCellOccupants.

**`_resolveCoinStep(player)`** ‚Äî `player.coins += 1`, broadcastPlayers.

**green_shell** ‚Äî set `pendingShellChoice = true`, broadcastPlayers, return (don't recurse).

### Resume after blocking states
- `discardCard` handler: if `pendingDiscard === 0 && pendingItems.length > 0` ‚Üí resume
- `shellChoice` handler: if `pendingItems.length > 0` ‚Üí resume

### Guards
Add `if (player.pendingItems.length > 0) return;` to: `endTurn`, `buyCard`, `playCard`.

### Clear on turn end
In `_endTurnAndAdvance` and `_endTurnForFinishedPlayer`: `player.pendingItems = [];`

## Slice 2: Message format changes

### `cardPlayed` ‚Äî simplify
Remove `droppedBanana` and `coinGained` fields. Just send `{ cardId, ...cardState }`.
The client reads item icons from the card's DOM element (`dataset.items`).

## Slice 3: Client player screen (animation only, no server dependency)

### Change `cardPlayed` handler
Read the card's items from `played.dataset.items` before removing it, then animate each item icon sequentially with staggered `setTimeout` calls (400ms apart). Icons: banana.svg, coin.svg, mushroom.svg, green_shell.svg.

Keep coin display update and shell modal from server response, delayed after item animations finish.
