# OOP Refactoring Plan

## Context

The codebase has one OOP example (`PlayerAvatar` in board.js) but the rest is procedural: `GameScene` is a 1260-line monolith, `player.js` has 17+ global variables with no classes, and `GameRoom.js` is a 1238-line single class. The goal is to identify cohesive objects/components that make the code more readable and maintainable, following the two patterns from CLAUDE.md:
- **Game Objects** (Phaser canvas): ES6 classes owning sprites, tweens, state
- **Components** (DOM UI): ES6 classes owning HTML, CSS, and behavior

## Recommended Extractions (prioritized by impact/effort)

### 0. Move PlayerAvatar to its own file
**File:** `client/src/board.js` → new `client/src/board_avatar.js`
**What:** Move the existing `PlayerAvatar` class to a separate file. Pure mechanical move + import update in board.js.
**Why:** Establishes the pattern — all Phaser game objects get their own file. Makes subsequent extractions consistent.
**Risk:** Very low. No logic changes.

### 1. CellGrid (server) — pure data structure
**File:** new `server/rooms/CellGrid.js` (separate file — pure data, no Room dependency)
**Encapsulates:** `cellOccupants` object + all 8 cell methods (`_cellOccupants`, `_addToCell`, `_removeFromCell`, `_countItemOnCell`, `_countShellsOnCell`, `_shellTypeOnCell`, `_hazardOnCell`, `_previousCell`) + `cells`/`prevCell` maps
**Why:** Most-called utilities in GameRoom, zero game-logic dependencies, completely self-contained. Safest extraction, enables unit testing without a Colyseus room. ~60 lines out of GameRoom.
**Risk:** Low. Mechanical `this._cellOccupants(id)` → `this.grid.occupants(id)` replacements.

### 2. CellItemSprites (board.js) — Phaser game object
**File:** new `client/src/board_items.js` (separate file)
**Encapsulates:** 4 sprite Maps (`bananaSprites`, `shellSprites`, `redShellSprites`, `permacoinSprites`) + `_inflightShells`/`_dustCloudCells` Sets + `_syncItemSprites` + item portions of `tweenCellLayout`/`snapCellLayout` + `createPermacoinSprites`/`repositionPermacoinSprites`
**Why:** Largest cohesive cluster in GameScene (~200 lines). Removes 6 Maps and 2 Sets from GameScene's constructor. Makes `tweenCellLayout` and `snapCellLayout` single-purpose (avatars only).
**Risk:** Medium. `animateShellThrow` and `animateItemHit` reach into sprite maps — expose `popSprite(cellId, type)` for that.

### 3. AnimationQueue (board.js) — utility class
**File:** new `client/src/board_animations.js` (separate file, could house both AnimationQueue and ShellAnimator)
**Encapsulates:** `_cellOccupantsQueue`, `_processingQueue` flag, `_enqueueAnimation`, `_processNextCellOccupants`
**Why:** Generic scheduling pattern (enqueue, process serially with delays) interleaved with animation dispatch. ~30 lines, very high clarity gain.
**Risk:** Very low. Pure scheduling logic.

### 4. ShellAnimator (board.js) — Phaser game object
**File:** `client/src/board_animations.js` (same file as AnimationQueue — both handle animation concerns)
**Encapsulates:** `animateShellThrow` (~120 lines, the single largest method), `animateItemHit` (~70 lines), `_spawnHitStars`, `_spawnDarkMushroom`, `_spawnDustCloud`
**Why:** Most complex methods in GameScene. The three `_spawn*` helpers are only called from animation code. Isolating makes these auditable.
**Risk:** Low-medium. Needs references to CellItemSprites and avatars Map. Best done after extractions 2 and 3.

### 5. CardDragHandler (player.js) — DOM component
**File:** new `client/src/player_drag.js` (separate file — follows existing split pattern: player_cards.js, player_buy.js, player_pile_modal.js)
**Encapsulates:** `addDragListeners` function (~100 lines), `getPointerPos` helper, and the `playing`/`animating`/`pendingDiscards`/`pendingShellChoice` state it manages
**Why:** First DOM component extraction, establishes the pattern for player.js. Complex touch/mouse event management with closure-heavy design currently closing over module globals. Owns the drag lifecycle.
**Risk:** Low. Self-contained UI interaction, no game logic.

### 6. DeckManager (server) — pure logic
**File:** new `server/rooms/DeckManager.js` (separate file — pure card logic, no Room dependency)
**Encapsulates:** `_createDeck`, `_createRiverDecks`, `_shuffle`, `_drawCards`, `_cardState` + test injection hooks (`_testDeck`, `_testRiverDecks`)
**Why:** Card/deck logic is fully independent from cell, turn, or player management. `_drawCards` with reshuffle logic benefits from isolation.
**Risk:** Low-medium. Methods mutate player objects by reference (fine in JS).

## Considered but NOT recommended

- **Player.js setupRoom as a class:** It's a wiring function, not a component. Making it a class just renames globals to `this.` without reducing complexity. Extract cohesive pieces first (CardDragHandler, etc.) and the wiring shrinks naturally.
- **Server TurnManager/PhaseManager:** `_startGame`, `_advanceTurn`, `_endTurnAndAdvance`, `_endGame` are deeply coupled to `this.players`, `this.ranking`, `this.phase`, and message sending. Too many dependencies for the benefit.
- **Server ShellResolver:** While complex, the shell methods need access to grid, players, messaging, and card state simultaneously. Extracting would require passing 5+ dependencies, creating a "manager" that's just a reshuffled version of the same coupling.
- **GameScene sub-scenes:** Phaser scene composition adds architectural complexity (inter-scene communication) not justified at this codebase size.

## Implementation order

Each extraction is one commit (edit → test → commit):

0. **PlayerAvatar** — move to own file, establish pattern
1. **CellGrid** — safest, zero UI risk
2. **CellItemSprites** — biggest readability win on client
3. **AnimationQueue** — small, clean, sets up #4
4. **ShellAnimator** — depends on #2 and #3
5. **CardDragHandler** — first DOM component
6. **DeckManager** — lower urgency but clean

## Verification

After each extraction:
- `npm test -w server` (server tests)
- `npm test -w client` (client tests)
- Manual play-through: create game, join 2 players, play a full round with card play, shell throws, buying from river, and verify animations work correctly
