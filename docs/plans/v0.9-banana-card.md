# Banana Card — Implementation Plan

## Context

The game currently has 8 identical `move_forward_1` cards per player. We're adding 2 banana cards (`banana_move_forward_1`) to each deck, bringing the total to **10 cards** (8 move + 2 banana). The banana card drops a banana on the player's current cell, then moves them forward 1. When any player lands on a cell with banana(s), they must discard one card per banana (player chooses). Bananas can stack on the same cell.

## Design Decisions

- **Server is the authority**: the client only sends `playCard { cardId }`. The server inspects the card type, decides to drop a banana, moves the player, detects banana hits, etc. The client never decides game logic — it only renders what the server tells it.
- **Player chooses** which card to discard (drag to play zone).
- **Bananas stack**: multiple bananas can exist on one cell (counter per cell).
- **Empty hand on banana hit**: server auto-draws cards first, then player discards.

## New/Modified Messages

| Message | Dir | Payload | Purpose |
|---------|-----|---------|---------|
| `cardPlayed` (mod) | S→C | `+ droppedBanana: cellId \| null` | Server informs client a banana was dropped (for animation) |
| `bananas` (new) | S→All | `{ cellId: count }` map | Server broadcasts banana positions |
| `bananaHit` (new) | S→C | `{ cellId, count, mustDiscard, autoDrawn? }` | Server tells player they hit banana(s), must discard |
| `discardCard` (new) | C→S | `{ cardId }` | Player's only action: choosing which card to discard |
| `cardDiscarded` (new) | S→C | `{ cardId, remaining, ...cardState }` | Server confirms discard, tells client how many left |

**Key principle**: the client sends only `playCard` and `discardCard`. All game logic (banana drop, banana hit detection, auto-draw) happens server-side. The server responses (`cardPlayed`, `bananaHit`, `cardDiscarded`) inform the client what happened so it can animate.

## Critical Files

- `server/rooms/GameRoom.js` — game logic, state, messages
- `client/src/player.js` — player hand, drag, play zone, animations
- `client/src/board.js` — board display (Phaser)
- `client/src/style.css` — play zone styling

## Slices

### Slice 1 — Backend: banana cards in the deck

**File**: `server/rooms/GameRoom.js`

- `_createDeck()`: 8× `move_forward_1` + 2× `banana_move_forward_1` (10 cards total), then `this._shuffle()`.
- `playCard` handler: add `banana_move_forward_1` to the move-forward condition so it also advances the player.

No new behavior yet — banana cards just move forward like normal cards.

### Slice 2 — Frontend player: banana card asset

**File**: `client/src/player.js`

- Add `banana_move_forward_1: "/card - banana and move forward.svg"` to `CARD_ASSETS`.

Everything else (draw animation, drag, render) works automatically via `CARD_ASSETS[card.type]`.

### Slice 3 — Backend: banana storage, drop, and broadcast

**File**: `server/rooms/GameRoom.js`

All logic is server-side. The client still just sends `playCard { cardId }`.

- `onCreate()`: init `this.bananas = {}` (object mapping cellId → count).
- In `playCard` handler: **the server** checks `card.type === "banana_move_forward_1"`, and if so, increments `this.bananas[player.cellId]` **before** moving the player.
- Add `broadcastBananas()`: `this.broadcast("bananas", this.bananas)`. Call it after banana drop.
- Call `broadcastBananas()` in `onJoin` for new clients to get current state.
- The server adds `droppedBanana: player.cellId` to the `cardPlayed` response (only for banana cards) so the client knows to animate a banana throw.

### Slice 4 — Frontend board: display bananas on cells

**File**: `client/src/board.js`

- `preload()`: load `banana` SVG.
- `constructor`: add `this.bananaSprites = new Map()` (cellId → array of Phaser images).
- Listen for `"bananas"` message from server: create/destroy banana sprites per cell to match counts.
- Position bananas at cell centers, slightly offset. Render behind helmets (lower depth).
- Handle resize: reposition banana sprites in `onResize()`.

### Slice 5 — Frontend player: banana throw animation on play

**Files**: `client/src/player.js`, `client/src/style.css`

- In `cardPlayed` handler: if server included `data.droppedBanana`, create `<img src="/banana.svg">` at play zone center, animate it upward with CSS (translateY -200px + fade out over ~600ms), remove after.
- CSS: `.banana-throw` keyframe animation.

### Slice 6 — Backend: banana hit + forced discard logic

**File**: `server/rooms/GameRoom.js`

All detection and enforcement is server-side:

- After moving player in `playCard`, **the server** checks `this.bananas[player.cellId]`.
- If bananas present:
  - Store count, remove bananas from that cell, call `broadcastBananas()`.
  - Set `player.pendingBananaDiscards = count` on the player state.
  - If `player.hand.length === 0`: server auto-draws cards (reuses draw logic internally), includes `autoDrawn` data in the `bananaHit` response.
  - Server sends `bananaHit` to client: `{ cellId, count, mustDiscard: count, autoDrawn? }`.
- Add `discardCard` message handler:
  - Guard: `player.pendingBananaDiscards > 0`.
  - Server removes card from hand, pushes to discard.
  - Decrements `player.pendingBananaDiscards`.
  - Server sends `cardDiscarded`: `{ cardId, remaining: player.pendingBananaDiscards, ...cardState }`.
- Guard `playCard` and `drawCards`: server blocks these if `pendingBananaDiscards > 0`.

### Slice 7 — Frontend player: banana hit discard UI

**Files**: `client/src/player.js`, `client/src/style.css`

The client reacts to server messages — no game logic decisions:

- Add `pendingDiscards = 0` flag.
- On `bananaHit` (from server):
  - **Crash animation first**: create a banana `<img>` above the viewport, animate it crashing down into the play zone center (translateY from -100% to center, with a slight bounce/scale effect, ~500ms). This visually communicates "you hit a banana."
  - After crash animation completes:
    - If `autoDrawn` present: play draw animation (reuse `animateDrawCards` logic).
    - Set `pendingDiscards = data.count`.
    - Replace play zone content: banana image + "Banana! Drag a card here to discard it."
    - Add `.banana-hit` class to play zone (yellow/orange border).
- Modify drag `onEnd`: if `pendingDiscards > 0`, send `discardCard` instead of `playCard`.
- On `cardDiscarded` (from server):
  - Remove discarded card from hand DOM, recompute fan.
  - Animate clone to discard pile, update pile counts.
  - If `data.remaining === 0`: restore play zone to default, remove `.banana-hit` class, set `pendingDiscards = 0`.
  - If `data.remaining > 0`: update label with remaining count.

## Verification

1. Start server + client. Two players join.
2. Player A draws cards — sees banana cards mixed in hand.
3. Player A plays a banana card — server drops banana, client sees banana thrown upward; board shows banana on that cell.
4. Player B moves to the banana cell — server detects hit, sends `bananaHit`; player sees banana in play zone with "Banana! Drag a card here to discard it."; board banana disappears.
5. Player B drags a card to play zone — sends `discardCard`, server confirms, UI restores.
6. Test stacking: play 2 banana cards on same cell, next player must discard 2 cards.
7. Test empty hand: player with empty hand hits banana, server auto-draws, then player discards.
