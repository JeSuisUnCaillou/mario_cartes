# v0.6 - Persist Player Data

## Context

Currently, each page reload creates a new player. The backend ties all player data to the WebSocket `sessionId`, which changes on every connection. Players lose their position and appear as new entries on the board.

The goal is to decouple player identity from connection identity so that reconnecting players recover their position and name.

## Architecture

Two maps in `GameRoom`:

- **`this.players`** (new) — `Map<playerId, { playerId, name, cellId, connected }>`. Persistent source of truth. Entries are never deleted during a game session.
- **`this.clientsInfo`** (existing, repurposed) — `Map<sessionId, { type, playerId? }>`. Connection registry only.

Player identification: client generates a UUID on first visit, stored in `localStorage` under `playerId:<gameId>`. Sent in join options on every connection.

Broadcast: `broadcastPlayers()` iterates `this.players` (not `clientsInfo`), includes a `connected` boolean.

## Slices

### Slice 1 — Backend: add `players` map and reconnect logic

**File: `server/rooms/GameRoom.js`**

- `onCreate`: add `this.players = new Map()`.
- `onJoin(client, options)`:
  - If player with known `playerId` → reconnect: mark `connected: true`, link sessionId.
  - If new player → `crypto.randomUUID()`, create entry in `this.players` with `cellId: 1`, send `playerId` back to client via `client.send("playerId", playerId)`.
  - Store `{ type, playerId }` in `clientsInfo`.
- `onLeave(client)`:
  - If player: set `connected: false` in `this.players`. Do NOT delete from `this.players`.
  - Delete from `clientsInfo`.
- Message handlers (`ping`, `changeName`): resolve via `clientsInfo[sessionId].playerId → players[playerId]`.
- `broadcastPlayers()`: iterate `this.players.values()`, emit `{ playerId, name, cellId, connected }`.

### Slice 2 — Player client: generate/store UUID, display cellId, skip form on reconnect

**File: `client/src/player.js`**

- `initPlayer`: check `localStorage` for `playerId:<gameId>` + `playerName`. If both exist, skip name form → `startGame(gameId, name, existingPlayerId)`.
- `startGame`: send `playerId` in join options. Listen for `"playerId"` message to store UUID. Listen for `"players"` message to display own `cellId`.
- Add `<p id="cell-info">` to the game screen template.

### Slice 3 — Board client: use `playerId` as key, render disconnected players

**File: `client/src/board.js`**

- Key `helmets` and `nameLabels` maps by `playerId` instead of `sessionId`.
- No `type` filter needed (broadcast only contains players).
- Disconnected players: `setAlpha(0.5)` on helmet and label.
- Remove cleanup that destroys helmets for missing players (all players are always broadcast).

## Verification

1. Start server, open board, open player on phone.
2. Join as player → see helmet on board at cell 1, see "Cell: 1" on phone.
3. Click ping → cell advances on both board and phone.
4. Reload player page → should skip name form, rejoin at same cell, helmet stays solid on board.
5. Close player tab → helmet goes semi-transparent on board.
6. Reopen player page → helmet goes solid again, cell unchanged.
7. Open second player → both appear independently on board.
