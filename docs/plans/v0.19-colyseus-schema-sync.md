# v0.19 — Migrate to Colyseus Schema State Sync

## Problem

The game uses manual `broadcast()` / `client.send()` for all state synchronization. This causes:

1. **Missed updates**: If a client has a brief network blip, it misses broadcasts and shows stale state (wrong positions, wrong phase, etc.) until the next broadcast of that type happens to fire.
2. **No delta compression**: Every broadcast sends the full payload (all players, all cell occupants), even if only one field changed.
3. **No built-in reconnection state**: When a client reconnects, it only gets state from the `onJoin` broadcasts. Any state change that happened between disconnect and reconnect is lost if no broadcast fires after reconnect.

## Solution

Migrate public game state to Colyseus's built-in schema-based state synchronization. Colyseus schemas:
- Auto-sync to all clients with **delta compression** (binary-encoded, only sends what changed)
- **Queue patches** so reconnecting clients receive all missed state changes
- Work with `allowReconnection()` to preserve sessions across brief disconnects

### What goes in schema vs. stays as messages

The project is on **Colyseus v0.15**, which constrains the API (`defineTypes`, no `@view()` for private state).

**Schema state** (auto-synced to all clients):
- `phase` — game phase string
- `currentRound` — round number
- `activePlayerId` — whose turn it is
- `players` — MapSchema of PlayerSchema (public fields only: name, cellId, connected, handCount, ready, coins, lapCount, pendingShellChoice, finished)
- `ranking` — ArraySchema of RankEntrySchema (playerId, name, rank)
- `rivers` — ArraySchema of RiverSchema (id, cost, slots, deckCount)
- `cellOccupants` — MapSchema of CellOccupantsSchema (ArraySchema of strings per cell)

**Keep as targeted messages** (private per-player data, no `@view()` in v0.15):
- `cardsDrawn` — hand cards, draw/discard counts, shuffle info
- `cardPlayed` — played card confirmation with card state
- `cardDiscarded` — discard confirmation with remaining count
- `cardBought` — purchase confirmation with card state
- `discardHit` — hit notification with discard requirements
- `playerId` — assigned player ID on first join
- `gameAlreadyStarted` — rejection message

**Keep as broadcast messages** (transient animation events, not state):
- `shellThrown` — shell throw animation data (from/to cell, hit result)
- `itemHitBoard` — item collision animation data (type, player, cell)

### Board animation queue

The board currently queues `cellOccupants` messages to animate movements sequentially. With schema sync, state changes arrive as `onChange` callbacks. The board will:
1. Listen to `players` MapSchema changes and `cellOccupants` changes via Colyseus callbacks
2. Queue those changes in `_cellOccupantsQueue` the same way it does today
3. The state object always reflects the latest truth, but visual animations play out in order

### Schema patch batching & async item resolution

**Problem**: Colyseus batches schema patches per tick (`patchRate` = 50ms default). `_processNextItem` is synchronous/recursive — if a player has items `["mushroom", "banana"]`, all `broadcastCellOccupants()` calls happen in the same call stack. With schema sync, only the final state would be sent, losing intermediate animation states (e.g., the mushroom move animation before the banana drop).

Same issue inside `_resolveMushroomStep`: the move (line 641) and banana/shell hit (line 663/680) broadcast cellOccupants twice synchronously.

**Solution**: Convert `_processNextItem` and `_resolveMushroomStep` to async chains using `this.clock.setTimeout()`:

- `_processNextItem`: replace the recursive call (line 632) with `this.clock.setTimeout(() => this._processNextItem(player), 0)` so each item resolution lands in a separate event loop tick → separate schema patch.
- `_resolveMushroomStep`: when a banana/shell hit occurs, split into two ticks:
  1. First tick: move player, sync state
  2. Second tick (via `this.clock.setTimeout`): send `itemHitBoard` broadcast, remove banana/shell, sync state, then continue to next item

When no hit occurs, `_resolveMushroomStep` continues immediately (no delay needed).

This changes `_resolveMushroomStep` from a void return to a callback-based or continuation-based pattern — it calls `_processNextItem` itself after the delay instead of returning to the caller.

### Reconnection

Use `allowReconnection()` in `onLeave` to hold the client's spot for 60 seconds. During that window, Colyseus buffers state patches. On reconnect, the client receives all missed patches automatically — no manual re-broadcast needed.

## Slices

### Slice 1: Define schema classes + wire up `this.setState()`

**Server changes:**

1. Create `server/rooms/schema.js` with schema classes (Colyseus 0.15 `defineTypes` API):
   - `PlayerSchema` (name, cellId, connected, handCount, ready, coins, lapCount, pendingShellChoice, finished)
   - `RankEntrySchema` (playerId, name, rank)
   - `RiverSlotSchema` (id, items) — items as a JSON string since card items are variable-length arrays
   - `RiverSchema` (id, cost, deckCount, slots as ArraySchema of RiverSlotSchema)
   - `CellOccupantsSchema` (entries as ArraySchema of string)
   - `GameState` (phase, currentRound, activePlayerId, players MapSchema, ranking ArraySchema, rivers ArraySchema, cellOccupants MapSchema)

2. In `GameRoom.onCreate()`, call `this.setState(new GameState())` and initialize default values.

3. Keep all existing plain JS properties (`this.players` Map, `this.cellOccupants` object, etc.) as the **server-side source of truth** for game logic.

4. Add a `_syncState()` method that copies from plain JS properties into the schema state (one-way sync, server logic → schema). Call it at every site where `broadcastPlayers()` / `broadcastGameState()` / `broadcastCellOccupants()` was called. Keep the old broadcast calls too for now (dual mode).

5. Convert `_processNextItem` from synchronous recursion to async:
   - Replace `this._processNextItem(player)` (line 632) with `this.clock.setTimeout(() => this._processNextItem(player), 0)`
   - This ensures each item step lands in a separate tick → separate schema patch

6. Convert `_resolveMushroomStep` to async when a hit occurs:
   - Move step: sync state immediately (move animation)
   - Hit step: `this.clock.setTimeout(() => { send itemHitBoard, remove item, sync state, continue }, 0)`
   - Change the control flow so `_resolveMushroomStep` calls `_processNextItem` itself after the hit delay, instead of returning to the caller's recursive loop
   - No-hit case: continue synchronously (no delay needed)

**No client changes yet** — the old `room.onMessage()` handlers still work alongside schema sync. This slice is purely additive on the server.

**Tests**: Existing server tests should still pass (broadcast messages still sent). Add tests verifying `room.state` schema reflects correct values after game actions. Test that async `_processNextItem` produces intermediate states visible across ticks.

### Slice 2: Client — board listens to schema state

**Client board changes (`board.js`):**
- Import Colyseus schema callbacks (`room.state.players.onAdd`, `room.state.players.onChange`, etc.)
- Replace `room.onMessage("players", ...)` with schema player callbacks → feed into existing `updateInfoBarPlayers()` and `_cellOccupantsQueue`
- Replace `room.onMessage("gameState", ...)` with `room.state.onChange` for phase/round/activePlayer → feed into existing `updateBoardGameState()`
- Replace `room.onMessage("cellOccupants", ...)` with schema cellOccupants callbacks → feed into existing `enqueueCellOccupants()`
- Keep `room.onMessage("shellThrown", ...)` and `room.onMessage("itemHitBoard", ...)` as-is (animation events)
- Rivers: listen to `room.state.rivers` changes → call `renderRivers()`

**Key concern**: The board animation queue. When schema state changes, the callback fires with the new state. We need to convert that into the same format the queue expects. Build a small adapter that converts schema callbacks into the plain objects the existing rendering code consumes.

**Tests**: Board function tests (pure functions) should be unaffected. Manual test: verify board animations still play correctly.

### Slice 3: Client — player listens to schema state

**Client player changes (`player.js`):**
- Replace `room.onMessage("gameState", ...)` with `room.state.onChange` for phase/round/activePlayer
- Replace `room.onMessage("players", ...)` with `room.state.players.onChange` for ready status sync
- Rivers: listen to `room.state.rivers` changes for buy modal
- Keep all private card message handlers (`cardsDrawn`, `cardPlayed`, `cardDiscarded`, `cardBought`, `discardHit`) as-is

**Tests**: Player function tests (pure functions) should be unaffected. Manual test: verify all player actions work correctly.

### Slice 4: Remove old broadcast methods + add `allowReconnection`

**Server changes:**
- Remove `broadcastPlayers()`, `broadcastGameState()`, `broadcastCellOccupants()` methods (they're now replaced by schema sync)
- Remove `room.onMessage("players/gameState/cellOccupants", ...)` handlers on clients (already replaced in slices 2-3)
- Add `allowReconnection(client, 60)` in `onLeave` for player clients — Colyseus will buffer patches during the disconnection window
- Keep the existing reconnect-by-playerId logic as a fallback for longer disconnections (> 60s)

**Tests**: Verify reconnection behavior: disconnect a client, change state, reconnect → client should have latest state. Existing tests updated to not rely on broadcast messages for the migrated state fields.

### Slice 5: Clean up debug mode

- Update `_debugSetGameState` and `_debugGetState` to work with schema state
- `_fullState()` can now read from `this.state` schema instead of manually assembling the object
- Update board debug panel if it relies on specific message formats

## Risks & mitigations

| Risk | Mitigation |
|------|------------|
| Schema has max 64 fields per class | Our schemas are well under this limit. GameState has ~7 top-level fields. |
| CellOccupants is a nested structure (MapSchema of ArraySchema) | Colyseus 0.15 supports this. Wrap per-cell arrays in a CellOccupantsSchema. |
| Board animation queue depends on discrete messages | Convert schema onChange callbacks into the same queued format. State is always latest; animations queue visually. |
| Schema patches batched per tick — synchronous broadcast chains lose intermediate states | Convert `_processNextItem` and `_resolveMushroomStep` to async with `this.clock.setTimeout()` between steps. Each step lands in a separate tick → separate patch. |
| River slots contain card items (variable arrays) | Serialize items as JSON string in the schema. Client deserializes. |
| v0.15 has no `@view()` for private state | Keep private card data as targeted `client.send()` messages. Only public state goes in schema. |
| Existing tests use `room.onMessage` to verify broadcasts | Update tests to check `room.state` schema values instead. |

## Out of scope

- Upgrading to Colyseus v0.16+ (would unlock `@view()` for private state, but is a separate migration)
- Moving private card data into schema (blocked by v0.15, no per-player filtering)
- Changing game logic — this is purely a sync layer migration
