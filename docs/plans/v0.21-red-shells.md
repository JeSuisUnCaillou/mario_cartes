# v0.21 - Red Shell Implementation Plan

## Context

Adding `"red_shell"` -- a homing variant of the green shell. Unlike green shells which check only the immediate adjacent cell, a red shell travels cell by cell in the chosen direction (forward/backward) until it hits a target (player, banana, or green shell) or loops back to the thrower (14-cell circuit), in which case it's destroyed with no effect. Red shells never stay on cells as occupants.

Red shells already exist in `decks.yaml` (rivers 2 and 3), and `assets/red_shell.svg` exists (served via Vite publicDir).

## Key Design Decisions

1. **`pendingShellType` on player**: When a shell item is processed, store `player.pendingShellType = "red_shell"` (or `"green_shell"`). The `shellChoice` handler reads this to dispatch to the correct resolve function. Added to `_cardState()` so the client knows which modal title to show (and handles reconnection).

2. **Red shells never become cell occupants**: The red shell always finds a target -- if no other player, banana, or green shell is in the way, it will loop back and hit the thrower themselves. No `"red_shell"` in `cellOccupants`, no red shell sprites on the board, no `_resolveMushroomStep` changes.

3. **`_resolveRedShell` as separate method**: Walks cell by cell using `next_cell`/`prevCell`, collecting the path. Stops at the first cell with a hittable target. Same hit priority as green shell (players > bananas > green shells).

4. **`shellThrown` message extended**: Adds `shellType: "green_shell"|"red_shell"` and `path: number[]` (cells traversed, for red shell animation). Green shell broadcasts also get `shellType` for consistency.

5. **Multi-cell animation**: Client tweens the red shell smoothly through all cells in `path` without stopping at intermediate cells -- only stopping at the final target cell to play the hit effect. The tween passes through each waypoint fluidly (~200ms per cell). Animation queue delay scales with path length.

## Slices

### Slice 1: Server -- `pendingShellType` and red_shell item processing

**`server/rooms/GameRoom.js`:**
- `_cardState()` (line 62): add `pendingShellType: player.pendingShellType`
- `_processNextItem()` (line 816): update green_shell block to also set `player.pendingShellType = "green_shell"`; add `else if (item === "red_shell")` block identical to green_shell but with `pendingShellType = "red_shell"`
- `shellChoice` handler (line 384): after getting direction, check `player.pendingShellType` -- if `"red_shell"`, call `_resolveRedShell(player, client, direction)` instead of `_resolveShell`. Reset `player.pendingShellType = null`
- Add `shellType: "green_shell"` to all 4 `shellThrown` broadcasts in `_resolveShell()` (lines 136, 150, 163, 175)

### Slice 2: Server -- `_resolveRedShell` traversal logic

**`server/rooms/GameRoom.js`:**
- Add `_resolveRedShell(thrower, throwerClient, direction)`:
  - Walk up to 14 cells in the chosen direction using `next_cell`/`prevCell`
  - At each cell, check occupants with same priority as green shell: other players (excluding thrower) > bananas > green shells
  - Since the shell always starts from an adjacent cell, it won't reach the thrower's cell until step 14 (full loop). At that point, if no other target was found, hit the thrower themselves. So: exclude thrower from player filter for the first 13 steps, include them on step 14 (or equivalently: if the current cell is the thrower's cell AND no non-thrower players are there, hit the thrower)
  - On hit: apply effect (discard for player, remove for banana/shell), broadcast `shellThrown` with `shellType: "red_shell"`, `path: [cellIds...]`, and hit info
