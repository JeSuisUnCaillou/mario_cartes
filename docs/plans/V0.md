# V0 Implementation Plan — Mario Cartes

## Context

This plan implements the V0 described in `docs/shaping/V0 - Initial pitch.md`. The goal is to lay down the minimal working components of the racing card game: a board screen (shared big screen), a player screen (mobile phone), and a Colyseus WebSocket backend managing game state. V0 focuses on correctness and testability over polish. No database, no persistence.

**Decisions locked in:**
- Monorepo: `client/` + `server/` + `shared/` in one repo
- Plain JavaScript (no TypeScript)
- Vitest for tests
- Track: simple rectangular loop, ~20 cells, rendered with Phaser graphics

## Slices

### Slice 0 — Repo Scaffold
Sets up the monorepo. Both `server` and `client` start without errors. Nothing renders yet.
- Root `package.json` with `npm workspaces`, `concurrently` for `npm run dev`
- `server/`: Colyseus + Express, listens on `PORT`
- `client/`: Vite + Phaser, logs "client alive" in console
- `shared/constants.js`: empty shell
- `.gitignore`, `Procfile`

**Verify:** `npm install` + `npm run dev` → server on 2567, Vite on 5173, both respond.

---

### Slice 1 — Shared Constants + Track Layout
Establishes the canonical data shapes everything else builds on.
- `shared/constants.js`: `CARD_TYPES`, `CELL_TYPES`, `PHASES`, `MAX_PLAYERS=8`, `LAPS_TO_WIN=3`
- `server/game/track.js`: `TRACK` — flat array of 20 `{ id, type, x, y }` cells. Cell 0 = `START_FINISH`. Cart position is an integer index; advancing N cells = `(pos + N) % TRACK.length`.
- `server/test/track.test.js`: 20 cells, cell 0 is `START_FINISH`, no duplicate IDs.

**Verify:** `npm test` passes in `server/`.

---

### Slice 2 — Deck Logic
Pure deck functions, fully tested.
- `server/game/deck.js`:
  - `buildStartingDeck()` → 7 cards (4× forward1, 2× banana+forward1, 1× forward2)
  - `shuffle(array)` → Fisher-Yates, new array
  - `drawCards(drawPile, discardPile, n)` → `{ drawn, newDrawPile, newDiscardPile }`, handles reshuffle-from-discard
- `server/test/deck.test.js`: correct distribution, draw 5 leaves 2, empty draw reshuffles discard.

**Verify:** `npm test` passes. Reshuffle edge case explicitly tested.

---

### Slice 3 — Game Rules (Card Resolution)
The heart of the game. Pure function, no Colyseus.
- `server/game/rules.js`:
  - `resolveRound(players, cardChoices, bananaPositions, trackLength)` → `{ players, bananaPositions, events[] }`
  - Processes players in turn order: move cart, place banana if applicable, detect banana landing (discard random card), detect lap completion
  - `events` = animation queue: `{ type, playerId, from, to, lapsCompleted, ... }`
- `server/test/rules.test.js`: forward 1/2, banana placement, banana landing penalty, lap detection, turn order respected.

**Verify:** `npm test` passes. Banana dropped by player A is landable by player B in same round.

---

### Slice 4 — Turn Manager
FSM for game lifecycle and turn order rotation.
- `server/game/turnManager.js`:
  - `createTurnState(playerIds)` → `{ phase: WAITING, turnOrder: [], round: 0 }`
  - `startGame(state)` → shuffles initial turn order, phase → PICKING
  - `recordChoice(state, playerId, card)` → when all players picked, phase → RESOLVING
  - `finishResolution(state, updatedPlayers)` → rotate turn order (first → end), round++, phase → PICKING or GAME_OVER
- `server/test/turnManager.test.js`: all-must-pick invariant, correct rotation, game over trigger.

**Verify:** `npm test` passes. 3-round simulation produces correct turn order.

---

### Slice 5 — Colyseus Room
Wires deck + rules + turnManager into a live game room.
- `server/rooms/GameRoom.js`:
  - `onCreate`: init game logic state (plain JS, not Colyseus schema)
  - `onJoin`: create player, build deck, draw 5 cards; private hand stored in `Map<sessionId, privateState>` (NOT in schema)
  - `onMessage("start_game")`: host triggers start
  - `onMessage("pick_card")`: calls `recordChoice`; when all picked, calls `resolveRound`, broadcasts `events`, calls `finishResolution`
  - Schema (public): `players { position, laps, handSize, finished }`, `phase`, `round`, `bananaPositions`
  - Private hand sent via `this.send(client, "hand_update", { hand })`
- `server/index.js`: registers `GameRoom`

**Verify:** Colyseus playground at `localhost:2567/colyseus` — join room, `start_game`, `pick_card`, observe state updates.

---

### Slice 6 — Client Network Layer
Shared Colyseus client for both views.
- `client/src/network/colyseusClient.js`:
  - `joinAsBoard(gameUid)` → joins room with `{ type: "board" }`
  - `joinAsPlayer(gameUid, playerName)` → joins room with `{ type: "player" }`
- `client/src/main.js`: reads URL path, routes to board or player app, passes room handle

**Verify:** Board tab + player tab both join same room, both log state to console.

---

### Slice 7 — Board Screen: Track + Carts (Static)
Phaser renders the track and carts. No animation yet — carts teleport.
- `client/src/board/BoardGame.js`: Phaser.Game, 1280×720, `BoardScene`
- `client/src/board/scenes/BoardScene.js`: `create()` builds track, `update()` positions carts from state
- `client/src/board/objects/TrackRenderer.js`: 20 Phaser Graphics rectangles, labels, checkered cell 0
- `client/src/board/objects/CartSprite.js`: colored circle + player number, positions from `TRACK[index].{x,y}`

**Verify:** Rectangular loop visible, carts appear at cell 0 on game start.

---

### Slice 8 — Player Screen: Hand UI
Mobile card selection with plain DOM (no Phaser).
- `client/src/player/PlayerApp.js`: listens to `hand_update` + `phase_change`; renders hand, disables after pick
- `client/src/player/HandView.js`: `<div class="card">` per card, tap → `pick_card` message
- `client/src/player/StatusView.js`: lap counter, round number, "waiting..." text

**Verify:** Cards appear on mobile, tapping sends `pick_card`, cards disable after selection, hand refreshes after round.

---

### Slice 9 — Cart Movement Animation
Carts animate step-by-step along the track using Phaser tweens.
- `client/src/board/scenes/BoardScene.js`: `playEventQueue(events)` — sequential tween chain
- `client/src/board/objects/CartSprite.js`: `animateTo(cellIndex, onComplete)` via `scene.tweens.add`
- Banana placement (yellow circle) animates before affected cart moves through it

**Verify:** Two-player game — carts slide visibly, banana appears then cart reacts.

---

### Slice 10 — QR Code Join Flow
Board shows a QR code linking to the player URL.
- Add `qrcode` to `client/package.json`
- `client/src/board/objects/QROverlay.js`: DOM canvas overlay top-left, generates QR for `/game/:uid/player`

**Verify:** QR visible top-left, scanning on phone opens player URL for correct UID.

---

### Slice 11 — End Game Screen
Ranking screen when all players finish 3 laps.
- `client/src/board/scenes/ResultScene.js`: ranked player list, ex-aequo support, "Play Again" button
- `BoardScene.js`: watches `phase === GAME_OVER`, transitions to `ResultScene`
- `StatusView.js`: shows "You finished! Lap 3/3"

**Verify:** Simulate game to completion, correct rank order shown, ties handled.

---

### Slice 12 — Heroku Deployment
Single Heroku dyno serves both server and built client.
- `Procfile`: `web: node server/index.js`
- `server/index.js`: in `NODE_ENV=production`, serve `client/dist/` as static via Express
- Root `package.json`: `heroku-postbuild` → `npm run build --workspace=client`
- `client/vite.config.js`: `base: "/"`, output to `dist/`

**Verify:** `npm run build` → `client/dist/` exists. `NODE_ENV=production node server/index.js` serves everything on one port. `git push heroku main` deploys successfully.

## Testing Strategy

| Layer | Tool | Approach |
|---|---|---|
| `deck.js`, `track.js`, `rules.js`, `turnManager.js` | Vitest (server) | Pure functions — no Colyseus, no network. Fast, deterministic. |
| `GameRoom.js` | Vitest + `@colyseus/testing` | `ColyseusTestServer` simulates join/message sequences |
| Client components | Vitest + jsdom | `HandView`, `StatusView` with DOM assertions |
| Integration | Manual / Colyseus playground | Two browser tabs: board + player |

**Key architectural invariant**: `rules.js` is a pure function. `GameRoom.js` only calls it and projects results into Colyseus schema. This keeps the testable surface large and network-coupled code minimal.
